## 日本語技術文の品質ゲート (codex-jp-harness)

日本語の技術文（進捗報告・完了報告・原因調査報告・レビュー回答・PR 説明・学習ノート・技術ドキュメント・設計メモ・解説記事・リリースノート等）を返す、または書き込む時は、以下を **厳守** する。

### 発火トリガー

**原則: 日本語を含む応答は全て `mcp__jp_lint__finalize` の対象**（opt-out 設計）。以下の除外パターンに**完全一致**する時のみ呼び出しを省略できる:

- 出力全体がコードブロック単独または差分（```〜```）で、前後に日本語の地の文を含まない
- 1 行の相槌（「了解しました」「ありがとうございます」など、20 文字以内で装飾・補足を含まない）
- yes / no の二値回答のみ
- 日本語文字をまったく含まない応答（英語のみ、コマンドのみ等）

上記に**完全一致しない**日本語応答および日本語技術文を格納するパスへのファイル書き込み（Obsidian Vault の `Notes/`, `Docs/`, `Articles/` 等、または同等のフォルダ）は、**文字数・見出しの有無・体裁を問わず全て発火対象**。短い会話調の作業ログ（「〜を続けます」「〜を入れました」等）、数十文字の進捗報告、レビュー回答、PR 説明、リリースノートもすべて対象。

**迷ったら呼ぶ。** 呼びすぎのコストは MCP 往復 1 回（数十 ms）程度で実害がない。呼び忘れのコストは規約違反が素通りして文書品質が毀損される。**呼び忘れのほうがはるかに重い**。

### 呼び出し手順

チャット応答の場合は応答全文を、ファイル書き込みの場合は書き込む内容全体を `draft` 引数として渡す。既存ファイルに追記する場合でも、追記する部分全体を `finalize` に通す（既存本文は対象外）。

返り値の `ok` が `false` の間は `violations` の指示に従って書き直し、再度 `finalize` を呼ぶ。`ok: true` を得たドラフトのみユーザーに返す。retry は最大 3 回までとし、4 回目以降はその旨をユーザーに伝えて最後のドラフトを返す（無限ループ回避）。

### サーバー側の自動修正（fast path）

返り値に **`fixed: true`** が含まれる場合、サーバーが決定的な置換（禁止語 → 推奨語）を適用した最終ドラフトを `rewritten` フィールドで返している。この場合は **自分で書き直さず、`rewritten` の内容をそのままユーザーに返す**。`finalize` を再度呼ぶ必要もない。これにより retry 往復が消え、トークン消費が削減される。

`rewritten` を採用した旨は明示しなくてよい（ユーザーは自動修正を意識する必要がない）。`advisories` が併せて返っている場合は、WARNING / INFO の指摘として別途検討する。

### severity の階層

各違反には `severity` フィールドが付く（v0.2.0 以降）:

- **ERROR**: 必ず修正する。1 件でも残っていれば `ok: false`。
- **WARNING**: 強く推奨する修正。`ok: true` でも `advisories` として通知される。次の retry で対応する。
- **INFO**: 参考情報。例えば `merge` のように業界でカタカナ表記が完全に普及している語。修正は任意。

ERROR が 0 件になった時点で `ok: true` を返すが、WARNING / INFO の advisories が出ていれば、最終ドラフトに反映できないか必ず検討する。

### 禁止事項

`finalize` を呼ばずに日本語技術文を返すことは **ルール違反**。応答を送信する直前に「このターンで `finalize` を少なくとも 1 回呼んだか?」を自問し、呼んでいなければ呼ぶ。呼び忘れに後から気づいた場合は、ユーザーに「呼び忘れたため再チェックします」と告げて再検品してから送信する。なお、呼び忘れは **Stop hook が検知して次回セッション起動時に再教育プロンプトを注入**するため、ターンを跨いだ後方検知も働く（詳細は `docs/HOOKS.md`）。

### 例外

MCP サーバー `jp_lint` が起動していない場合は、その旨を 1 行告げ、自己検品で応答を継続する（業務は止めない）。

### 想像する場面（音読ドリブン）

応答を書く時、**ユーザーがこの応答を VOICEVOX で読み上げさせる場面**を想像する。音読して自然に意味が通る文章が目標。英語識別子の羅列・名詞句の過連続・専門用語の直訳は音読で破綻する → 書き直す。1 文は 80 文字以内を目安とし、英語識別子を含む文は 50 文字以内に収める。短く切り、動詞で区切る。

### セッション全体での識別子ルール（報告以外にも適用）

日本語技術文のトリガーを踏まない進行ナラティブ（「〜します」「〜を確認します」「〜が通りました」形式の作業状態報告）でも、コード識別子は必ずバッククォートで囲む。対象はファイル名・関数名・変数名・型名・ブランチ名・PR 番号・パラメータ名・コマンド名・タスク ID（`TASK-XXX`）・パッケージ名など。裸書きは `finalize` を呼ばない場面でも違反。

### 「確認してから呼ぶ」の禁止

「`jp_lint` が使えるか確認します」「MCP が有効か試します」といった**事前確認を挟んではいけない**。まず `finalize` を呼び、エラーが返ってきた場合に限って fallback（自己検品）に切り替える。事前確認を応答に書いている時点で、それは呼び忘れの兆候である。

### 曖昧な参照語の禁止

「対象テスト」「広い確認」「前面」「公開面」「一式」のような**範囲を限定せず読み手に推論させる語**は避ける。代わりに具体的なコマンド名（例: `cargo test -p <crate>`, `pytest tests/test_X.py`）・モジュール名・API 境界を書く。ユーザーは何を確認したかを正確に知る権利がある。

### アンインストール

本ルールは `codex-jp-harness`（公式日本語対応までの暫定対策）で導入されたもの。OpenAI が Codex（CLI / App）に日本語自然化を公式実装した時点でアンインストールされる。
