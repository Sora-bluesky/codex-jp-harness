   p. 日本語の技術進捗報告・完了報告・原因調査報告・レビュー回答・PR 説明を
      返す時は、以下を **厳守** する:

      **① 発火トリガー**: 応答に「やったこと」「確認結果」「次にやること」の
      見出しが現れる、または進捗・完了・原因・レビューの内容を含む時点で、
      そのドラフトは日本語技術報告である。トリガー条件を満たした瞬間から
      MCP ツール `mcp__jp_lint__finalize` の呼び出しは**必須**。

      **② 呼び出し手順**: ドラフト全文を `draft` 引数に渡す。返り値の `ok` が
      `false` の間は `violations` の指示に従って書き直し、再度 `finalize` を呼ぶ。
      `ok: true` を得たドラフトのみユーザーに返す。retry は最大 3 回までとし、
      4 回目以降はその旨をユーザーに伝えて最後のドラフトを返す（無限ループ回避）。

      **③ 禁止事項**: `finalize` を呼ばずに日本語技術報告を返すことは
      **ルール違反**。応答を送信する直前に「このターンで `finalize` を
      少なくとも 1 回呼んだか?」を自問し、呼んでいなければ呼ぶ。呼び忘れに
      後から気づいた場合は、ユーザーに「呼び忘れたため再チェックします」と
      告げて再検品してから送信する。

      **例外**: MCP サーバー `jp_lint` が起動していない場合は、その旨を 1 行告げ、
      自己検品で応答を継続する（業務は止めない）。

      **想像する場面 (音読ドリブン)**: 応答を書く時、**ユーザーがこの応答を VOICEVOX で
      読み上げさせる場面**を想像する。音読して自然に意味が通る文章が目標。英語識別子の
      羅列・名詞句の過連続・専門用語の直訳は音読で破綻する → 書き直す。1文は 80 文字
      以内を目安とし、英語識別子を含む文は 50 文字以内に収める。短く切り、動詞で区切る。

      **セッション全体での識別子ルール（報告以外にも適用）**: 日本語技術報告のトリガーを
      踏まない進行ナラティブ（「〜します」「〜を確認します」「〜が通りました」形式の
      作業状態報告）でも、コード識別子は必ずバッククォートで囲む。対象はファイル名・
      関数名・変数名・型名・ブランチ名・PR 番号・パラメータ名・コマンド名・タスク ID
      （`TASK-XXX`）・パッケージ名など。裸書きは `finalize` を呼ばない場面でも違反。

      **「確認してから呼ぶ」の禁止**: 「`jp_lint` が使えるか確認します」「MCP が有効か
      試します」といった**事前確認を挟んではいけない**。まず `finalize` を呼び、エラーが
      返ってきた場合に限って fallback（自己検品）に切り替える。事前確認を応答に書いて
      いる時点で、それは呼び忘れの兆候である。

      **曖昧な参照語の禁止**: 「対象テスト」「広い確認」「前面」「公開面」「一式」の
      ような**範囲を限定せず読み手に推論させる語**は避ける。代わりに具体的なコマンド名
      （例: `cargo test -p winsmux-core`, `pytest tests/test_rules.py`）・モジュール名・
      API 境界を書く。ユーザーは何を確認したかを正確に知る権利がある。

      本ルールは `codex-jp-harness`（公式日本語対応までの暫定対策）で
      導入されたもの。公式対応完了時には撤去される。
